# 技术详设：三类自动化打通 + 五端页面结构扫描

> 配套文档：本文是《[自动化打通与多端扫描方案](./自动化打通与多端扫描方案.md)》的**技术详设版**。
> 概览文档讲"做什么"，本文讲"怎么做"（架构设计级——含接口签名、表结构、队列架构、各端方案），但不含逐行业务代码。
>
> 状态：决策已确认，待评审后逐项建任务实施。

## 0. 决策记录（已确认）

| # | 决策项 | 最终结论 | 含义 |
|---|---|---|---|
| ① | 执行环境 | **独立执行机 + 任务队列** | 后端只派发，执行在独立 Runner 节点；后端机器不被浏览器/设备拖垮，可横向扩展 |
| ② | 设备来源 | **云真机为主 + 本地模拟器兜底** | Android 本地模拟器先跑通；iOS/鸿蒙/真机走云真机 |
| ③ | 鸿蒙范围 | **纯血鸿蒙 HarmonyOS NEXT（hypium）** | 用华为 hypium 框架，不当安卓测 |
| ④ | 小程序范围 | **仅微信小程序（minium）** | 走 MiniProgRunner，统一接入 Runner 层 |
| ⑤ | 一期范围 | **激进：五端架构全铺，执行按环境分批点亮** | 代码架构一次到位不留技术债；各端"真实执行"随环境就绪逐个开启 |

**激进路线的落地纪律**：架构层（Runner 接口、Dispatcher、队列、数据库）一次性支持全部六类执行（api/web/android/ios/harmony/miniprogram）；但每类 Runner 的真实实现，按环境就绪度分批交付，未就绪的回退到 MockRunner，保证平台始终可用。

## 1. 现状基线（实施必须基于的事实）

代码核对结论（来自 `backend/app/models/models.py` 等）：

- `TestCase.case_type`：当前枚举仅 `ui / api`。
- `TestCase.platforms`：字符串数组，**已含 `backend_api`** 标记后端接口；可承载 `web/android/ios/harmony/miniprogram`，无需新增字段即可扩展。
- `TestCase.script` / `script_status` / `is_automated`：脚本正文与状态已具备。
- 执行：`services/mock_runner.py` 是**假执行**（随机 pass/fail），是本次改造的核心替换对象。
- 脚本生成：`agents/script_generator.py` 已有 `api/web/app` 模板，但 `determine_script_type` **写死返回 `"web"`** —— 需改为按 `case_type`+`platforms` 真实路由。
- `PageStructureCache`：有 `base_url / url_pattern / dom_hash / regions / status`，**无 platform 维度**，唯一定位靠 `url_pattern`。
- `PageCacheDiff`：已有"执行中发现结构变化 → 排队 → 人工确认回写"机制，多端扫描应复用它。
- 扫描：`services/page_recorder.py` 走 `playwright codegen`，PC 视口写死 `1280,800`，仅 web。

> 设计原则：**最大化复用**。`TestResult` / 质量门禁 / 缺陷分析 / `PageCacheDiff` 全部保持不变，新增只在"执行"和"扫描"两层。

## 2. Runner 抽象层与 Dispatcher

### 2.1 分层

```
routers/executions.py
        │  POST /executions （创建执行，入队）
        ▼
ExecutionDispatcher          ← 决定每个用例走哪个 Runner
        │  按 (case_type, platforms) 路由
        ▼
TaskQueue（执行任务）         ← 派发到执行机
        ▼
┌───────────── Runner 节点（独立进程/机器）─────────────┐
│  BaseRunner                                          │
│   ├─ MockRunner        （兜底，环境未就绪时）          │
│   ├─ ApiRunner         httpx + pytest                │
│   ├─ WebRunner         Playwright + pytest           │
│   ├─ AppRunner         Appium（android/ios）          │
│   ├─ HarmonyRunner     hypium                        │
│   └─ MiniProgRunner    minium（微信）                 │
└──────────────────────────────────────────────────────┘
        │  回调：统一 RunOutcome
        ▼
回填 TestResult → 质量门禁 → 缺陷分析（均不变）
```

### 2.2 BaseRunner 接口（设计级签名）

```python
# services/runners/base.py
from dataclasses import dataclass

@dataclass
class RunOutcome:
    status: str            # passed/failed/skipped/error
    duration_ms: int
    error_message: str | None = None
    screenshot_url: str | None = None
    failure_type: str | None = None     # script_error/env_error/real_defect
    raw_report: dict | None = None      # pytest json-report 原文，便于回溯

class BaseRunner(ABC):
    platform: str          # api/web/android/ios/harmony/miniprogram
    requires_device: bool  # 是否需要设备（决定调度到哪类执行机）

    async def run(self, case: TestCaseDTO, ctx: RunContext) -> RunOutcome:
        workdir = self._prepare(case, ctx)     # 写脚本+依赖到隔离目录
        report  = await self._execute(workdir, ctx)  # 子进程跑 pytest --json-report
        return self._parse(report)             # 解析 → 统一 RunOutcome

    @abstractmethod
    def _prepare(self, case, ctx) -> Path: ...
    @abstractmethod
    async def _execute(self, workdir, ctx) -> dict: ...
    def _parse(self, report) -> RunOutcome:    # 默认解析 pytest-json-report，子类可覆写
        ...
```

要点：
- 所有 Runner 统一产出 `RunOutcome`，回填逻辑只写一份。
- `_execute` 统一用 `pytest --json-report`，差异收敛在 `_prepare`（装什么依赖、起什么会话）。
- `RunContext` 携带：执行机能力、设备 udid、base_url、登录态、页面结构缓存快照。

### 2.3 Dispatcher 路由规则

```python
def resolve_runner(case) -> str:
    if case.case_type == "api" or "backend_api" in case.platforms:
        return "api"
    for p in ("miniprogram", "harmony", "ios", "android", "web"):
        if p in case.platforms:
            return p
    return "web"   # 默认
```

- `EXECUTION_MODE=mock` 或目标 Runner 环境未就绪 → 一律回退 `MockRunner`（保证激进路线下平台不挂）。
- 路由结果写入 `TestResult.raw`/日志，便于排查"为什么走了某 Runner"。

### 2.4 脚本类型路由修正

`agents/script_generator.py::determine_script_type` 由写死 `"web"` 改为：

| case_type / platforms | script_type | 引擎 |
|---|---|---|
| `api` 或含 `backend_api` | `api` | httpx |
| 含 `web` | `web` | Playwright |
| 含 `android`/`ios` | `app` | Appium |
| 含 `harmony` | `harmony` | hypium |
| 含 `miniprogram` | `miniprogram` | minium |

`SYSTEM_TEMPLATES` 需新增 `harmony` / `miniprogram` 两套提示词模板。

## 3. 数据库改造

### 3.1 PageStructureCache 多端化（核心改动）

| 字段 | 动作 | 说明 |
|---|---|---|
| `platform` | **新增** `String(20)` | `web/android/ios/harmony/miniprogram`，默认 `web` |
| `page_key` | **新增** `String(500)` | 统一页面标识：web=URL路径，原生=Activity/Ability路由，小程序=page path |
| `raw_source` | **新增** `JSONB` 可空 | 原始结构快照（DOM片段/控件树XML/ArkUI树/WXML），便于回溯与 diff |
| `url_pattern` | **保留** | 兼容旧数据；web 端 `page_key` 可等于它 |
| `regions/dom_hash/status` | 不变 | 区域结构、哈希、状态语义全端通用 |
| 唯一键 | **改** | `(project_id, platform, page_key)` 取代原 `url_pattern` 单列定位 |

迁移策略（Alembic）：
1. 加列 `platform` 默认 `'web'`、`page_key`、`raw_source`。
2. 数据回填：存量行 `page_key = url_pattern`、`platform = 'web'`。
3. 建唯一索引 `(project_id, platform, page_key)`。
4. `page_recorder` 写入逻辑补 `platform='web'`、`page_key=路径`。

> `PageCacheDiff` 差异队列同样加 `platform`，全端复用"发现变化→排队→人工确认回写"，无需另造轮子。

### 3.2 执行/结果模型微调

- `Execution`：新增 `execution_mode`（mock/real）、`runner_node`（执行机标识），便于审计。
- `TestResult`：现有 `failure_type`（script_error/env_error/real_defect）已够用；`raw_report` 可塞进现有 `ai_diagnosis` 或新增 `raw` JSONB（二选一，建议新增以免语义混淆）。
- 不动 `case_type` 枚举值，靠 `platforms` 数组承载五端——**零破坏性**。

## 4. 任务队列与执行机架构（决策①）

```
后端(FastAPI)                          执行机(Runner Worker, 可多台)
   │ 创建 Execution(status=pending)
   │ 拆成 N 个 RunTask 入队 ───────────▶  消息队列(Redis/RQ 或 Celery)
   │                                         │ worker 领取 RunTask
   │                                         │ 选 Runner → run() → RunOutcome
   │  ◀─── 回调 /internal/run-callback ───────┘
   │ 回填 TestResult，累计 Execution 统计
   │ 全部完成 → 质量门禁 → 缺陷分析 → 飞书通知
```

- **队列选型建议**：一期 Redis + RQ（轻量，够用）；规模上来再换 Celery。
- **执行机分类**：`无设备机`（api/web）、`安卓机`、`iOS机`(Mac)、`鸿蒙机`、`小程序机`。RunTask 带 `requires` 标签，worker 按能力领取。
- **云真机接入**：AppRunner/HarmonyRunner 的 `_execute` 通过云真机平台 API 申请设备会话，udid 注入 `RunContext`。
- **隔离**：每个 RunTask 在独立临时目录跑，互不污染；失败截图回传对象存储，URL 落 `TestResult.screenshot_url`。
- **回退**：worker 探测到目标引擎不可用 → 标记 `env_error` 或交还队列降级 Mock（按配置）。

## 5. 五端：页面扫描 + 执行方案

每端两件事：**结构扫描**（喂 `PageStructureCache`）和**用例执行**（Runner）。

### 5.1 接口（api）
- **扫描**：无"页面"，但可做 **接口契约扫描**——导入 OpenAPI/Swagger/抓包 HAR，生成接口清单当"结构缓存"的接口版（可选，二期）。
- **执行**：`ApiRunner` = httpx + pytest，无环境依赖。**最先点亮**。

### 5.2 PC web（web）
- **扫描**：现有 `playwright codegen`（录制式）。增强点：视口可配（不再写死 1280×800）、写入时补 `platform='web'`。
- **执行**：`WebRunner` = Playwright + pytest。复用 Runner 层。

### 5.3 Android（android）
- **结构来源**：原生控件树 XML；**页面标识** = Activity 名（`driver.current_activity`）。
- **扫描**：
  - 模式A（一期）：appium-inspector / `driver.page_source` 录制 → 正则解析控件树（套用 `_parse_recorded_script` 思路）→ 入表 `platform='android'`。
  - 模式B（二期）：自动遍历——dump 页 → 提可点击元素 → 点击 → dump 新页 → 页面指纹去重防死循环。
- **执行**：`AppRunner`（android 分支）= Appium UiAutomator2 + pytest。本地模拟器先跑通。
- **定位优先级**：resource-id > accessibility-id > xpath。

### 5.4 iOS（ios）
- **结构来源**：XCUIElement 树；**页面标识** = 可见的导航标题 / VC 推断。
- **扫描**：同 Android 模式A/B，driver 换 XCUITest，`platform='ios'`。
- **执行**：`AppRunner`（ios 分支）= Appium XCUITest + WebDriverAgent。**依赖 Mac + Xcode**，走云真机。
- **定位优先级**：accessibility-id > predicate/class-chain > xpath。

### 5.5 纯血鸿蒙（harmony）
- **结构来源**：ArkUI 组件树；**页面标识** = Ability/Page 路由。
- **扫描**：`hdc` + hypium dump 组件树 → 解析 → 入表 `platform='harmony'`。生态较新，一期先做录制式 + 手动 dump。
- **执行**：`HarmonyRunner` = hypium（华为官方 UI 测试框架）+ pytest 风格用例。依赖鸿蒙真机/云真机。
- **风险**：hypium 资料少、API 仍在演进——一期目标"能跑通单条用例 + 能 dump 一个页面结构"即可，不追全覆盖。

### 5.6 微信小程序（miniprogram）
- **结构来源**：WXML（双线程，类 DOM）；**页面标识** = 小程序页面路由 path（如 `pages/index/index`）。
- **扫描**：minium 提供页面元素查询，遍历当前页可见组件 → 入表 `platform='miniprogram'`。
- **执行**：`MiniProgRunner` = minium（微信官方自动化 SDK）+ pytest。依赖微信开发者工具 + 测试号 + automator 端口。
- **定位**：minium 自有 selector（data 属性 / 文本 / xpath-like）。

### 5.7 各端能力矩阵

| 端 | platform 值 | 扫描工具 | 执行引擎 | 设备依赖 | 一期目标 |
|---|---|---|---|---|---|
| 接口 | `web`(api路由) | OpenAPI导入(可选) | httpx | 无 | ✅ 真实执行 |
| PC web | `web` | playwright codegen | Playwright | 无 | ✅ 真实执行 |
| Android | `android` | appium dump | Appium UiAutomator2 | 模拟器/真机 | ✅ 真实执行 |
| iOS | `ios` | appium dump | Appium XCUITest | Mac+真机/云 | 🟡 通路打通 |
| 鸿蒙 | `harmony` | hdc + hypium dump | hypium | 鸿蒙真机/云 | 🟡 单用例跑通 |
| 微信小程序 | `miniprogram` | minium 查询 | minium | 开发者工具+测试号 | 🟡 通路打通 |

> ✅=一期完整交付，🟡=一期打通通路（架构+至少一条用例），覆盖度二期补齐——符合"激进铺架构、执行分批点亮"。

## 8. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 鸿蒙 hypium 生态新、资料少 | P5 卡壳 | 一期只求单用例跑通+单页 dump，不追覆盖 |
| iOS 依赖 Mac+Xcode+WDA | 环境重 | 走云真机，本地不自建 |
| 小程序需测试号+automator端口 | 接入门槛 | 申请专用测试号，文档化 automator 启动步骤 |
| 五端并发把执行机打满 | 执行慢/排队 | 队列按设备类型分池，云真机按需扩容 |
| AI 生成脚本不可直接运行 | 真实执行报 script_error | 失败归类 `script_error` 走现有修复 Agent；保留 mock 兜底 |
| 自动遍历死循环 | 扫描跑飞 | 页面指纹去重 + 最大深度/步数上限 |

## 9. 一句话总结

- **打通** = BaseRunner 抽象层 + Dispatcher 按 `(case_type, platforms)` 路由 + 独立执行机/队列；六类 Runner 统一产出 `RunOutcome`，回填 `TestResult` 一份逻辑，质量门禁/缺陷分析零改动。
- **多端扫描** = `PageStructureCache` 加 `platform`/`page_key`/`raw_source`，各端用各自结构来源（DOM/控件树/ArkUI/WXML），复用现有 `PageCacheDiff` 差异确认机制；一期录制式，二期自动遍历。
- **激进落地纪律** = 架构一次铺全六端，真实执行按环境就绪分批点亮，未就绪回退 Mock，平台始终可用。
