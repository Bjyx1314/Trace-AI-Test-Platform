# 测试平台：三类自动化打通 + 多端页面结构扫描 方案

> 目标：把「接口自动化 / PC 自动化 / 移动端自动化」在平台内打通；并解决「页面结构提前扫描」在移动端（Android / iOS / 鸿蒙 / 小程序）的落地。
>
> 本文只谈方案与路线，不含代码改动。评审通过后再逐项建任务实施。

---

## 0. 现状盘点

| 模块 | 当前实现 | 问题 |
|---|---|---|
| 执行引擎 | `services/mock_runner.py` | **假执行**：随机 pass/fail，没真正跑脚本 |
| 脚本生成 | `agents/script_generator.py` | 已有 `api/web/app` 模板，但 `determine_script_type` 写死 `"web"` |
| 页面扫描 | `services/page_recorder.py` | 走 `playwright codegen`，**只支持 PC web**，视口写死 1280×800 |
| 页面缓存表 | `models.PageStructureCache` | 用 `url_pattern` 做 key，**没有 platform 维度**，移动端无 URL 无法入表 |
| 用例模型 | `TestCase.case_type` = `ui/api`，`platforms` 数组 | 维度够，但执行层没按它分流 |

**结论**：脚本「生成」层已区分三类，但「执行」层和「扫描」层尚未打通。

---

## 1. 打通的核心思路：统一调度，分类执行

打通 = **入口统一 + 结果统一 + 中间按类型路由到不同 Runner**。

```
                  ┌─────────────────────────────────────────────┐
  执行请求 ──────▶│           ExecutionDispatcher                │ 按 case_type/platform 路由
                  └───┬────────┬────────┬────────┬────────┬──────┘
                      │        │        │        │        │
                ┌─────▼──┐ ┌──▼───┐ ┌──▼────┐ ┌─▼─────┐ ┌▼────────┐
                │ApiRunner│ │WebRun│ │AppRun │ │Harmony│ │MiniProg │
                │ httpx   │ │Playw.│ │Appium │ │ hdc/  │ │ minium/ │
                │ +pytest │ │+pytest│ │+pytest│ │ DevEco│ │ 自动化  │
                └─────┬───┘ └──┬───┘ └──┬────┘ └─┬─────┘ └┬────────┘
                      └────────┴────────┴────────┴────────┘
                                统一回填 TestResult
```

### 1.1 三个不变量（打通的关键）
1. **统一用例模型**：所有端共用 `TestCase`，靠 `case_type` + `platforms` 区分。
2. **统一结果模型**：所有 Runner 产出收敛到同一张 `TestResult`，Dashboard / 质量门禁 / 缺陷分析无需改动。
3. **统一 Runner 接口**：`BaseRunner.run(case) -> TestResult`，子类各自封装执行引擎。

### 1.2 Runner 抽象层设计
```
BaseRunner(ABC)
  ├─ run(case) -> RunOutcome        # 统一入口
  ├─ _prepare()  生成/写脚本到临时目录
  ├─ _execute()  子进程跑 pytest（--json-report）
  └─ _parse()    解析报告 -> 统一结果

子类：
  MockRunner      （保留，开发态/演示用，配置开关切换）
  ApiRunner       httpx + pytest          —— 无环境依赖，最先落地
  WebRunner       Playwright + pytest     —— 已有 codegen 基础
  AppRunner       Appium + pytest         —— Android/iOS
  HarmonyRunner   hypium / hdc            —— 鸿蒙
  MiniProgRunner  minium(微信官方) / 自动化SDK —— 小程序
```

### 1.3 从「假执行」到「真执行」最小一步
接口端最容易先打通（无浏览器、无设备）：
1. AI 生成的脚本写临时目录
2. `pytest --json-report` 子进程执行
3. 解析 JSON 回填 `TestResult`
4. 配置开关 `EXECUTION_MODE=mock|real` 灰度切换

## 2. 页面结构提前扫描：多端统一设计

### 2.1 核心矛盾
PC web 用 DOM 扫描，但**其它端没有 DOM**，结构来源各不相同：

| 端 | 结构来源 | 定位方式 | 页面标识 | 扫描工具 |
|---|---|---|---|---|
| **PC web**（现有） | HTML DOM | CSS / role | URL | `playwright codegen` |
| **Android** | 原生控件树（XML） | resource-id / a11y-id / xpath | Activity 名 | Appium `page_source` / inspector |
| **iOS** | XCUIElement 树 | accessibility-id / xpath | ViewController | Appium（WebDriverAgent） |
| **鸿蒙(HarmonyOS)** | ArkUI 组件树 | id / key / type | Ability/Page 路由 | hypium + hdc dump |
| **小程序** | 双线程，类 DOM 的 WXML | 自定义选择器 | 小程序页面路由 path | minium / 小程序自动化 SDK |

### 2.2 数据模型改造（关键前置）
`PageStructureCache` 当前以 `url_pattern` 为 key，需扩展为多端通用：

```
新增字段：
  platform   : web | android | ios | harmony | miniprogram
  page_key   : 统一页面标识（web=URL，原生=Activity/路由，小程序=page path）
  raw_source : 原始结构快照（XML/JSON），便于回溯
保留：
  regions / dom_hash / page_name / status
唯一键：
  (project_id, platform, page_key)   ← 替代原来的 url_pattern
```

> 这样一张表同时承载五端的页面结构，扫描结果、用例关联、变更检测全部复用现有逻辑。

### 2.3 扫描两种模式（每端可选其一）

**模式 A — 录制式（对标现有 codegen，工作量小）**
人工操作一遍，工具录制后解析结构 → 入表。
- PC：已有（`playwright codegen`）
- Android/iOS：appium-inspector 录制 → 解析控件树
- 小程序：minium 录制 / 微信开发者工具自动化
- 鸿蒙：DevEco 录制 / hdc dump 手动触发
- 复用现有 `_parse_recorded_script` 的「按页分组 + 元素聚合」套路

**模式 B — 自动遍历式（真正的"提前自动扫描"，工作量大）**
启动应用 → dump 当前页结构 → 提取可点击元素 → 自动点击 → dump 新页 → 递归。
- 需处理：登录态、弹窗、死循环防护（页面指纹去重）、返回栈
- 建议二期再做，一期先用模式 A 把数据跑通

### 2.4 各端落地难度与依赖

| 端 | 模式A难度 | 模式B难度 | 环境依赖 | 建议 |
|---|---|---|---|---|
| PC web | ✅ 已有 | 中 | 无 | 已具备 |
| Android | 低 | 中 | Appium + SDK + 真机/模拟器 | 优先 |
| iOS | 中 | 高 | Mac + Xcode + WebDriverAgent | 次之 |
| 小程序 | 中 | 中 | minium + 微信开发者工具 + 测试号 | 次之 |
| 鸿蒙 | 中高 | 高 | DevEco + hypium + 鸿蒙设备 | 生态最新，放最后 |

## 3. 落地路线图（每步独立可交付）

> 原则：先地基（接口真执行）→ 再复用（PC 真执行）→ 再扩展（多端扫描）→ 最后补齐（多端执行）

| 阶段 | 任务 | 产出 | 依赖 |
|---|---|---|---|
| **P1** | Runner 抽象层 + `ApiRunner` 接口真实执行 | 接口用例真正跑起来，结果落 `TestResult` | 无 |
| **P2** | `WebRunner` PC web 真实执行 | 复用 Runner 层，Playwright 真跑 | P1 |
| **P3** | `PageStructureCache` 加 `platform`/`page_key` 字段 + 迁移 | 多端结构可入同一张表 | 无（可与 P1 并行） |
| **P4** | 移动端录制式扫描（Android 优先，模式 A） | Android 页面结构入表 | P3 |
| **P5** | `AppRunner` 移动端真实执行（Android） | Android 用例真跑 | P1, P4 |
| **P6** | 扩展 iOS / 小程序 / 鸿蒙（扫描 + 执行） | 全端覆盖 | P3, P5 |
| **P7**（可选） | 自动遍历式扫描（模式 B） | 真正的"提前自动扫描" | P4 |

## 4. 待决策项（实施前需确认）

1. **执行环境在哪跑**：平台后端本机？还是独立的执行机/CI Runner？（涉及设备接入、并发、隔离）
2. **真机 or 云真机**：自建设备农场 vs 接入云测平台（影响 AppRunner/HarmonyRunner 设计）
3. **鸿蒙范围**：是 HarmonyOS NEXT（纯血鸿蒙，用 hypium）还是兼容安卓的旧版？两者扫描方式不同。
4. **小程序范围**：仅微信小程序，还是含支付宝/抖音小程序？（不同的自动化 SDK）
5. **一期取舍**：是否先做「P1+P2+P3+P4(Android)」四步打底，移动端其余三端二期再上？

---

## 5. 一句话总结

- **打通** = 引入 Runner 抽象层，统一入口和结果，按 `case_type/platform` 路由到 Api/Web/App/Harmony/MiniProg 各自引擎；先把接口的「假执行」换成「真执行」当地基。
- **多端扫描** = 给 `PageStructureCache` 加 `platform` 维度，各端用各自的结构来源（DOM/控件树/WXML/ArkUI），一期走「录制式」复用现有解析套路，二期再上「自动遍历」。
